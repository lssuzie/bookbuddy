## 一、核心理念

### 一句话

> **让读书变成一件你想做的事，而不是你应该做的事。**

### 为什么会有这个东西

因为对自己有用。用户喜欢梁朝伟，如果用他的声音来读书，会更愿意听，也更容易把一本书听完。既然这个逻辑对一个人成立，那对很多人应该也成立。

### 核心洞察

**读书的障碍从来不是书本身，是"开始读"的那一步。**

传统链条：
```
想读 → 翻开书 → 枯燥 → 放弃
```

BookBuddy 的新链条：
```
想读 → 打开 → 听到喜欢的声音 → 愉悦 → 继续听
```

**门槛不在"读"，在"想读"。** 用情感连接降低认知门槛。

## 三、产品演化历程

### 阶段一：Agent Skill（起点）

两个独立的 Skill：

- `everything-to-podcast` — 文本清洗、分片、TTS、合并的流水线引擎
- `tony-leung-tts` — 梁朝伟声音克隆配置+参考音频

### 阶段二：统一引擎

把声音克隆注入 `generate_audio.py`，加 `--voice-clone` 参数，统一入口。

### 阶段三：编排层

创建 `tony-audiobook` Skill，三层架构：

```
编排层 (tony-audiobook)     ← 纯指令，无代码
  ├── 引擎层 (everything-to-podcast)  ← 开源
  └── 声源层 (tony-leung-tts)        ← 私有
```

三个分支：
- **分支A**：全文朗读 — 用户给文件，直接生成
- **分支B**：AI代读播客 — AI 先总结再读
- **分支C**：找书+朗读 — 搜索公版书下载后再转

### 阶段四：开源策略

意识到 `everything-to-podcast` 作为通用引擎可以开源，`tony-leung-tts`（含梁朝伟参考音频）不能开源。

开源版剥离为：引擎（通用）+ 用户自备声音（录音5秒即可）

### 阶段五：产品化 → BookBuddy

从"工具"升维为"产品"——不是文本转语音的工具，而是**用喜欢的声音读书的陪伴式阅读体验**。

未来形态：
```
BookBuddy
├── ⚙️ 引擎 (CLI) — 生成有声书
├── 🎨 书架页面 (Web) — 文字+音频同步高亮
├── 🌧️ 氛围音 — 雨声/森林/壁炉/海浪
└── 🤖 AI — 代读/总结/推荐
```

### 空白市场

**中文 + 声音克隆 + 有声书流水线 = 无人占领。** 没有竞品同时满足这四个条件。

### 从 kokoro-tts (⭐1611) 学到的

kokoro-tts 成功的原因：
1. **一条命令跑通** — `pip install kokoro-tts` 装完就能用
2. **零配置** — 自带模型，不需要 API Key
3. **README 第一眼就展示效果** — 有演示、有对比

对 BookBuddy 的启示：
- 减少外部依赖（当前依赖 MiMo API Key 是门槛）
- 降低开始使用的步骤（如果可能，提供免费试用额度）
- README 要有视觉冲击力（终端录屏 GIF）

## 五、架构决策

### 三层 Skill 模型

```
编排层 (tony-audiobook)     ← 纯指令，变更频繁
  ├── 引擎层 (everything-to-podcast)  ← 代码稳定
  └── 声源层 (tony-leung-tts)        ← 参考音频+配置
```

- 编排层不写代码，只有 SKILL.md
- 引擎层可开源给任何人用
- 声源层是私有的（含参考音频）

### 改名决策

`everything-to-podcast` → `bookbuddy`

原因：
1. "Podcast" 误导用户期待 AI 主持人节目，实际是逐字有声书
2. 0 Star 时改名成本为零
3. `bookbuddy` 传达的是"陪伴感"，不是"工具感"

### 拆与不拆

结论：**不拆**。一个 repo，多场景展示。

三个场景共用同一个引擎（清洗→分片→TTS→合并），拆成三个 repo = 同一份代码维护三遍。

## 六、伦理与版权

### 声音版权

- 工具是开源的，不包含任何名人声音
- 用户使用自己的声音（录 5 秒）或已获授权的参考音频
- 在 README 中明确伦理承诺

### 书籍版权

- 公版书（Gutenberg/古籍）— 可自动下载
- 商业书 — 用户自己提供文件
- 不爬取盗版网站

### 伦理承诺原文

> **伦理承诺**
> 此工具是个人工具，不是内容分发平台。
> - **声音**：请只使用你自己的声音，或你已获授权的声音参考
> - **内容**：请只朗读你有权访问的文本
> - **禁止**：不要用于冒充他人，不要用于商业盈利
> 工具本身是中立的。你用它做什么，是你的选择。

## 八、开源策略

### 当前状态

- 只有一个开源项目：`everything-to-podcast`（将改名为 `bookbuddy`）
- 已在 GitHub 推送
- 0 Star，0 Issue

### 起步建议

1. **不要急于商业化** — 先让 100 个人用起来
2. **README 要有视觉冲击力** — 终端演示 GIF
3. **写教程** — 《如何用 30 秒克隆自己的声音做有声书》
4. **回复每个 Issue** — 开源声誉 = 回复率 × 代码质量
5. **等有人问"怎么付费"再考虑商业化**

### 商业化时机

当以下信号出现：
- 有人主动问"怎么付费"
- 有企业用户询问授权
- 维护成本开始超过个人能承受的范围

在此之前，保持开源，积累声誉。
